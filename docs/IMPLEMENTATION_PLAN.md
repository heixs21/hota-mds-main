## 任务状态划分

| 分类 | 任务 |
| --- | --- |
| 现在可以先做 | M0 接口契约与范围固化；M1 工程骨架与基础部署；M2 后台基础能力与主数据基础模型；M3 标准数据模型、缓存层与 mock；M4 基于 mock/cache API 的左右屏大屏页面；M6 数据源健康状态与异常兜底基础能力 |
| 必须等待外部系统资料 | M5 中的 OPCUA / Modbus 点表接入；M5 中的 SAP RFC 字段和连接方式接入；M5 中的排产系统数据库表结构接入；M5 中的能耗数据库表结构接入；真实数据联调与 UAT 中的真实数据验收 |
| 适合前后端并行 | M2：后端做主数据、配置、映射模型/API，前端做后台维护页面；M3：后端做 mock/cache API，前端基于 mock 契约开发展示组件；M4：后端稳定展示 API，前端实现左右屏 UI、轮播、全屏适配；M6：后端做健康状态和兜底逻辑，前端做后台健康页和大屏异常不白屏展示；M5：后端/collector 按数据源适配，前端做真实数据展示校验 |
| 高风险任务 | 甘特图：涉及产线分组、30 日窗口、可见截取、自动滚动、延期/暂停状态样式；延期预测：涉及剩余数量、有效产能优先级、计划完工时间、未开工订单不判断等口径；数据源映射：涉及设备、产线、订单、物料、区域等多系统编码统一 |

## 推荐执行顺序

| 顺序 | 阶段 | 原因 |
| --- | --- | --- |
| 1 | M0 | 先锁定一期前段边界，避免把报修、3D、内部报表带入核心交付 |
| 2 | M1 | 先建立工程、容器、环境变量和基础启动能力，后续开发才有承载基础 |
| 3 | M2 | 后台配置、主数据、编码映射是数据接入和展示配置的基础 |
| 4 | M3 | 外部资料未齐时，先做标准模型、缓存层和 mock，保障前端可以并行推进 |
| 5 | M6 基础能力 | 异常兜底是大屏稳定展示底线，建议在真实接入前先完成基础能力 |
| 6 | M4 | 基于 mock/cache API 跑通左右屏展示链路，尽早暴露甘特图、全屏适配、轮播问题 |
| 7 | M5 | 等外部系统点表、字段、表结构、连接方式具备后再接真实数据，避免空接或返工 |
| 8 | UAT | 在真实数据、异常兜底、双屏展示均完成后，进行验收和上线准备 |
