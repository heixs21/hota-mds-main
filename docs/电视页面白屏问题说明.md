# 电视页面白屏问题说明

本文档记录和泰智造数屏系统在 **小米 HyperOS 智能电视浏览器**（及同类 Android TV WebView）上访问时的白屏、布局异常问题，包括原因分析、当前解决方案、部署注意事项、方案优缺点及后续优化建议。

**验证环境参考：**

| 项目 | 值 |
| --- | --- |
| 设备 | 四核处理器，2GB 运行内存 |
| 系统 | Xiaomi HyperOS（OS1.0.89.0.PSTAATV） |
| 浏览器 | 内置 WebView（内核偏旧，约 Chrome 79 以下） |
| 访问方式 | 局域网 IP + 前端静态服务 |

---

## 1. 问题概览

| 现象 | 类型 | 说明 |
| --- | --- | --- |
| 整页纯白，`#root` 无任何内容 | **JS 白屏** | JavaScript 未成功执行或启动阶段崩溃 |
| 有「数据加载中…」或「展示页暂时不可用」 | **数据/接口问题** | JS 正常，非本文「白屏」范畴 |
| 页面能打开，但内容挤在左侧、右侧大片空白 | **布局/viewport 问题** | 视口宽度被错误上报 |
| 布局正常，但文字、卡片、图标整体偏小 | **远距观看/UI 缩放问题** | 需单独放大 UI |

---

## 2. 白屏问题

### 2.1 原因分析

#### 2.1.1 直接使用 Vite 开发模式（主要原因）

电视访问 `http://<IP>:5173`（`npm run dev`）时，浏览器需加载 Vite 开发客户端：

- `@vite/client`
- `@react-refresh`

上述脚本使用 **可选链 `?.` 等现代 JavaScript 语法**。旧版 TV WebView 不支持，会在解析阶段抛出：

```text
Uncaught SyntaxError: Unexpected token .
  @vite/client:66
  @react-refresh:612
```

错误发生在 **React 应用挂载之前**，因此 `#root` 为空，页面表现为白屏。

> **结论：** 白屏根因是 **Vite dev 基础设施与 TV WebView 不兼容**，不是后端 API 无数据。项目内 `ScreenDisplay` 已有加载态与异常兜底，接口失败不会导致完全空白。

#### 2.1.2 其他可能原因（本次未触发，但需知晓）

| 原因 | 说明 |
| --- | --- |
| 访问 `localhost` / `127.0.0.1` | 电视上的 localhost 指向电视自身，非开发机 |
| 生产包语法仍过新 | 未设置 `build.target` 时，构建产物可能含 TV 不支持的语法 |
| 内存不足（2GB） | 复杂页面可能导致 WebView 进程被系统杀掉 |
| 历史问题：`React is not defined` | 早期未接入 `@vitejs/plugin-react` 导致；当前已修复 |

### 2.2 当前解决方式

#### （1）电视端禁止使用 dev，改用 preview / 生产构建

```bash
# 推荐：本地联调电视
cd frontend
npm run preview:tv

# 或 Docker 生产部署
docker compose up --build
```

| 方式 | 电视访问地址 | 端口 |
| --- | --- | --- |
| `npm run preview:tv` | `http://<局域网IP>:4173/screen/{areaCode}/left` | 4173 |
| Docker Compose | `http://<局域网IP>:3000/screen/{areaCode}/left` | 3000 |
| ❌ 不可用于电视 | `http://<IP>:5173`（`npm run dev`） | 5173 |

#### （2）构建目标降级（`frontend/vite.config.js`）

```javascript
build: {
  target: "chrome63",
}
```

将业务代码中的 `?.`、`??` 等语法编译为旧 WebView 可执行的 ES 版本。

> **注意：** 此配置 **无法** 降级 Vite dev 客户端本身，故 dev 模式在电视上仍不可用。

#### （3）preview 模式 API 代理

`vite.config.js` 中 `preview.proxy` 将 `/api` 转发至后端（默认 `http://localhost:8000`），与 dev 行为一致。使用 preview 时需保证后端已启动。

#### （4）临时调试：`window.onerror`（可选，排查后可删）

在 `frontend/index.html` 中加入了内联错误捕获脚本，JS 报错时会在页面上显示红底错误信息，便于无开发者工具的电视现场排查。

```javascript
<!-- 临时调试：电视白屏时在页面上显示 JS 错误，排查完请删除 放在<div id="root"></div>之前 -->
    <script>
      (function () {
        var box = document.createElement("div");
        box.id = "hota-debug-errors";
        box.style.cssText =
          "display:none;position:fixed;inset:0;z-index:99999;padding:24px;" +
          "background:#1a0000;color:#ffb4b4;font:14px/1.5 monospace;" +
          "white-space:pre-wrap;overflow:auto;word-break:break-all;";
        document.body.appendChild(box);

        function show(title, detail) {
          box.style.display = "block";
          box.textContent += (box.textContent ? "\n\n" : "") + title + "\n" + detail;
        }

        window.onerror = function (message, source, lineno, colno, error) {
          show("[onerror]", [
            String(message),
            source ? source + ":" + lineno + ":" + colno : "",
            error && error.stack ? error.stack : "",
          ].filter(Boolean).join("\n"));
          return false;
        };

        window.addEventListener("unhandledrejection", function (event) {
          var reason = event.reason;
          show("[unhandledrejection]", reason && reason.stack ? reason.stack : String(reason));
        });
      })();
    </script>
```



### 2.3 白屏方案优缺点

| 方案 | 优点 | 缺点 |
| --- | --- | --- |
| **preview / Docker 生产包** | 根本解决 dev 脚本兼容问题；性能更好 | 每次改代码需重新 build |
| **`build.target: chrome63`** | 业务 JS 兼容旧 WebView | 不解决 dev 模式；包体积可能略增 |
| **`window.onerror` 调试** | 现场可见错误，无需 ADB | 仅辅助排查，非正式功能；应排查后删除 |
| **继续用 dev (:5173)** | PC 开发体验好 | **电视必然白屏**，不可用 |

## 3. 部署注意事项

### 3.1 网络与地址

1. **必须使用局域网 IP**，例如 `http://192.168.x.x:4173`，不要使用 `localhost`。
2. 确认电视与服务器在同一网段，防火墙放行对应端口（4173 或 3000，以及后端 8000 若直连）。
3. PC 开发可继续用 `npm run dev`（5173），**电视验收必须走 preview 或 Docker**。

### 3.2 构建与发布流程

```bash
# 前端电视联调
cd frontend
npm install
npm run preview:tv

# 正式部署
docker compose up --build
```

修改 `index.html`、`styles.css`、`ScreenDisplay.jsx` 等前端文件后，电视端需 **重新 build** 才能生效。

### 3.3 后端依赖

- preview 模式下 `/api` 由 Vite 代理到 `VITE_PROXY_TARGET`（默认 `http://localhost:8000`）。
- Docker 模式下 nginx 将 `/api` 代理至 `backend:8000`。
- 大屏路由示例：`/screen/05/left`、`/screen/05/right`（`05` 换为实际区域编码）。

### 3.4 与项目设计的关系

README / PRD 要求：**数据源异常时不白屏**。当前 `ScreenDisplay` 在首次加载失败时显示 `ScreenFallback`，加载中显示 `ScreenLoading`。电视上的 **完全空白** 应优先排查 JS 加载与兼容性，而非后端数据。

### 3.5 电视推荐访问 URL

```text
# 联调 preview（默认 1×）
http://<局域网IP>:4173/screen/{areaCode}/left

# 电视大屏展示（推荐加 scale）
http://<局域网IP>:4173/screen/{areaCode}/left?scale=1.35

# Docker 生产
http://<局域网IP>:3000/screen/{areaCode}/left?scale=1.35
```

将 `{areaCode}` 换为实际区域编码（如 `05`）。倍率可按现场观看距离微调，建议范围 1.25～1.4。

## 4. 布局不一致问题

### 4.1 现象

viewport 修正前，电视浏览器出现：

- 内容集中在页面 **左侧**
- **右侧 30%～40% 为空白黑边**
- 与 PC 同 URL 的多列网格布局差异明显

修正后布局正常，但文字与控件相对 PC **偏小**（远距观看问题，见第 5 节）。

### 4.2 原因分析

#### 4.2.1 视口宽度被错误上报（主要原因）

`index.html` 原配置：

```html
<meta name="viewport" content="width=device-width, initial-scale=1.0" />
```

小米 TV WebView 常将 `device-width` 报为 **960px 或 1280px**（即使物理屏幕为 1080p/4K），且 **不会自动拉伸至全屏宽度**。页面按 ~960px 排版，在更宽屏幕上左对齐显示，右侧留空。

PC Chrome 的视口宽度通常为真实窗口宽度（1920px+），`@media (min-width: 1600px)` 等断点正常生效，故布局不同。

#### 4.2.2 CSS 现代特性在旧 WebView 上失效（次要）

| 特性 | 最低 Chrome 版本 | 失效影响 |
| --- | --- | --- |
| `min()` / `max()`（CSS） | 79 | 网格列定义 `minmax(min(100%, 280px), 1fr)` 失效 |
| `clamp()` | 79 | 字号回退异常 |
| `:has()` | 105 | `html:has(.screen-left)` 高度/overflow 规则不生效 |
| `100dvh` | 108 | 有 `100vh` 兜底，影响较小 |

### 4.3 当前解决方式

#### （1）方案 A：强制 viewport 物理宽度（已实施）

位置：`frontend/index.html` `<head>` 内，在页面渲染前执行：

```javascript
var w = Math.max(window.screen.width || 0, window.screen.height || 0, 1920);
meta.setAttribute(
  "content",
  "width=" + w + ", initial-scale=1.0, maximum-scale=1.0, user-scalable=no",
);
```

按屏幕物理宽高较大值设置 layout viewport，下限 1920px，使 TV 与 PC 1920 屏使用相近的逻辑宽度。

#### （2）方案 1：UI 缩放（已实施，替代原方案 2 zoom）

**触发条件：** 访问 `/screen/*` 时，`ScreenDisplay.jsx` 为 `<html>` 添加 `screen-display` class，并设置 CSS 变量 `--screen-ui-scale`。

**默认倍率：** `1`（与 PC 原尺寸一致，不放大）。

**URL 可调：** 通过查询参数 `scale` 指定放大倍率，例如 `?scale=1.35`。合法范围 **0.5～3.0**；缺失、非法或超出范围时回退为 **1**。

**实现位置：**

- `frontend/src/ScreenDisplay.jsx` — 解析 `scale` 参数并写入 `document.documentElement.style`
- `frontend/src/styles.css` — 基于 `--screen-ui-scale` 缩放 rem 与关键 px 尺寸

```css
html.screen-display {
  --screen-ui-scale: 1;
  font-size: calc(16px * var(--screen-ui-scale));
}
```

```javascript
// ScreenDisplay.jsx（逻辑摘要）
const scale = resolveScreenUiScale(); // 读 URL ?scale=，默认 1
document.documentElement.style.setProperty("--screen-ui-scale", String(scale));
```

**访问示例：**

```text
# 默认 1×
http://<IP>:4173/screen/05/left

# 电视远距观看推荐（现场曾验证 1.3～1.35 较合适）
http://<IP>:4173/screen/05/left?scale=1.35
```

**缩放机制：**

- 所有 **rem** 字号随 `html` 根字号同比变化。
- 关键 **px** 间距（padding、gap、logo、按钮、卡片等）通过 `calc(... * var(--screen-ui-scale))` 放大。
- 对 TV 不支持的 **`clamp()`** 提供固定字号 fallback（标题、KPI、监控卡片等）。

**说明：** 曾临时使用 **方案 2（`zoom: 1.3`）** 验证效果，后迁移为方案 1；倍率由硬编码 1.3 改为 **默认 1 + URL 可配**，便于 PC 预览与电视现场分别调参。

### 4.4 布局方案优缺点

| 方案 | 优点 | 缺点 |
| --- | --- | --- |
| **方案 A：viewport 脚本** | 改动小；一次修复全页宽度 | 依赖 `screen.width` 上报；极端设备可能需写死 1920/3840 |
| **方案 1：根字号 + px 缩放** | 标准 CSS；比 zoom 更清晰；支持 URL `?scale=` 配置 | 部分 px 写死样式需单独维护 override 列表 |
| **方案 2：zoom（已弃用）** | 一行代码、全元素同比放大 | 非标准；可能影响全屏/点击区域；高倍缩放略糊 |
| **方案 3：调整 initial-scale** | 实现简单 | 会改变逻辑宽度，易再次引发布局错乱，不推荐单独使用 |
| **写死 `@media` 大屏样式** | 精确控制 | PC 与 TV 同宽度时无法区分；维护成本高 |

## 5. 字号偏小问题（布局正常后）

### 5.1 原因

1. viewport 固定为 1920px 后，逻辑分辨率与 PC 1920 显示器相同，但 **电视观看距离更远**（约 3～5 米），相同字号主观上更小。
2. 样式大量使用 0.72rem～1.1rem 的小字号及 `clamp(..., vw, ...)`，旧 WebView 不支持 `clamp` 时回退字号偏小。

### 5.2 当前参数

| 项目 | 值 | 说明 |
| --- | --- | --- |
| `--screen-ui-scale` 默认值 | `1` | 不传 `scale` 时与 PC 原尺寸一致 |
| 电视推荐值 | `1.3`～`1.35` | 远距观看现场验证较合适，通过 URL 指定 |
| URL 参数 | `?scale=<number>` | 例：`?scale=1.35`；合法范围 0.5～3.0 |
| 作用范围 | 仅 `html.screen-display` | 后台 `/admin/*` 不受影响 |

### 5.3 URL 缩放参数说明

| 参数 | 类型 | 默认 | 说明 |
| --- | --- | --- | --- |
| `scale` | number | `1` | 大屏 UI 整体缩放倍率 |

**示例：**

| URL | 效果 |
| --- | --- |
| `/screen/05/left` | 1×，默认 |
| `/screen/05/left?scale=1.3` | 1.3× 放大 |
| `/screen/05/left?scale=1.35` | 1.35× 放大 |
| `/screen/05/left?scale=abc` | 非法值，回退 1× |
| `/screen/05/left?scale=5` | 超出上限 3.0，回退 1× |

> **部署建议：** 电视浏览器可将主页书签设为带 `?scale=1.35`（或现场测定值）的完整 URL，避免每次手动调参。

## 8. 快速排查流程

```text
电视异常
  │
  ├─ 整页空白？
  │    ├─ 是 → 是否访问 :5173 dev？ → 改为 preview:tv 或 Docker :3000
  │    │         仍有错 → 查看 onerror 红字 / ADB 远程调试 Console
  │    └─ 否 → 是否有「加载中/不可用」？ → 查网络、/api 代理、后端健康
  │
  ├─ 内容挤在左侧？
  │    └─ 确认 index.html viewport 脚本已部署并重新 build
  │
  └─ 布局正常但字太小？
       └─ 在 URL 追加 ?scale=1.35（或 1.3）；默认无参数为 1× 不放大
```
