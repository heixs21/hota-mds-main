# 和泰智造数屏系统 PLAN

## 1. 里程碑总览

建议按以下顺序推进：

1. 里程碑 M0：需求冻结与接口契约准备。
2. 里程碑 M1：工程骨架与基础部署。
3. 里程碑 M2：后台基础能力与主数据。
4. 里程碑 M3：标准数据模型、缓存层与 mock 数据。
5. 里程碑 M4：一期前段大屏页面。
6. 里程碑 M5：真实数据源接入。
7. 里程碑 M6：数据源健康状态与异常兜底。
8. 里程碑 M7：一期后段报修接入。
9. 里程碑 M8：一期后段低优先级 3D 仿真。
10. 里程碑 M9：二期内部 Web 报表预研。

## 2. M0：需求冻结与接口契约准备

目标：

- 固化一期前段范围。
- 明确前段验收不包含报修和 3D 仿真。
- 定义标准 API 契约与数据结构草案。

任务：

- 整理 SPEC、PLAN、STATUS、HANDOFF。
- 定义设备、产量、排产、能耗的标准返回结构。
- 定义数据源健康状态结构。
- 定义左右屏配置结构。

验收标准：

- 一期前段验收边界明确。
- 关键数据对象和接口方向明确。
- 报修、3D 仿真、内部报表边界明确。

依赖关系：

- 依赖 PRD 与需求澄清结论。

## 3. M1：工程骨架与基础部署

目标：

- 搭建 Django + DRF、React、collector、Docker 的基础工程骨架。

任务：

- 初始化后端工程。
- 初始化前端工程。
- 初始化 collector 服务目录。
- 准备 Docker Compose 或部署模板。
- 连接独立 MySQL。
- 配置基础环境变量。

验收标准：

- backend 容器可启动。
- frontend/nginx 容器可启动。
- collector 容器可启动。
- backend 可连接独立 MySQL。
- 本地或测试环境可访问基础页面。

依赖关系：

- 依赖 M0。
- 不依赖外部系统点表或表结构。

## 4. M2：后台基础能力与主数据

目标：

- 支持系统后台登录、配置和主数据维护。

任务：

- 管理员登录。
- 设备台账。
- 产线台账。
- 区域台账。
- 订单与物料基础数据模型。
- 编码映射维护。
- 展示内容配置。
- 左右屏轮播配置。
- 页面模块开关。
- 数据源配置。
- 阈值与预测参数配置。
- 单日有效工作时间配置。

验收标准：

- 管理员可登录后台。
- 可维护主数据。
- 可维护编码映射。
- 可配置左右屏展示与轮播。
- 可配置数据源连接信息，但连接信息不写死在代码里。

依赖关系：

- 依赖 M1。
- 是 M3、M4、M5 的基础。

## 5. M3：标准数据模型、缓存层与 mock 数据

目标：

- 在外部系统资料未提供前，先完成标准数据模型与缓存层，支持前端并行开发。

任务：

- 设备状态快照模型。
- 产量统计快照模型。
- 近 8 小时产量趋势模型。
- 排产甘特图模型。
- 延期预测结果模型。
- 区域能耗快照模型。
- 数据源健康状态模型。
- mock 数据生成。
- 定时任务框架。
- 最近成功更新时间记录。

验收标准：

- 后端可返回设备、产量、排产、能耗 mock API。
- 前端可基于 mock API 开发。
- 后端具备缓存数据读取接口。
- 后台能看到 mock 数据源健康状态。

依赖关系：

- 依赖 M2。
- 不依赖真实点表和外部数据库结构。

## 6. M4：一期前段大屏页面

目标：

- 完成左右屏页面展示，先基于 mock 或缓存 API 跑通展示链路。

任务：

- `/screen/left` 页面。
- `/screen/right` 页面。
- 左屏欢迎展示。
- 左屏设备运行概览。
- 左屏产量执行概览。
- 左屏近 8 小时产量趋势。
- 左屏区域能耗概览。
- 左屏报修占位区。
- 右屏未完工订单甘特图。
- 右屏延期风险说明。
- 右屏 3D 仿真占位区。
- 全屏适配。
- 自动轮播。
- 左右屏独立配置读取。

验收标准：

- 左右屏可通过独立 URL 打开。
- 左右屏可全屏展示。
- 左右屏可按后台配置轮播。
- 报修区域显示占位。
- 3D 仿真区域显示占位。
- 数据异常时页面不白屏。

依赖关系：

- 依赖 M3。
- 可与 M5 前期准备并行。

## 7. M5：真实数据源接入

目标：

- 将一期前段核心数据源接入真实系统。

任务：

- OPCUA / Modbus TCP 设备与机械手数据接入。
- SAP RFC 订单与物料信息接入。
- 排产系统数据库接入。
- 能耗数据库接入。
- 外部系统编码映射到本系统主编码。
- 定时拉取任务。
- 缓存写入。
- 数据异常记录。

建议接入顺序：

1. 排产系统数据库。
2. SAP RFC。
3. 能耗数据库。
4. OPCUA / Modbus TCP。

验收标准：

- 设备、产量、排产、能耗均可真实接入。
- 数据按默认 5 分钟周期更新。
- 数据可写入缓存层。
- 大屏展示真实数据。
- 外部系统异常时保留最近一次成功数据。

依赖关系：

- 依赖 M2、M3。
- 依赖外部系统点表、字段、连接方式整理完成。
- 可按资料提供情况调整接入顺序。

## 8. M6：数据源健康状态与异常兜底

目标：

- 保证外部参观大屏稳定展示。

任务：

- 数据源健康状态页。
- 数据源最近成功更新时间。
- 数据是否过期判断。
- 后台异常提示。
- 大屏最近一次成功数据展示。
- 接口错误兜底。
- 日志记录。

验收标准：

- 后台可查看设备、产量、排产、能耗数据源状态。
- 后台可查看最近成功更新时间。
- 后台可识别数据过期。
- 大屏不显示过期提示。
- 大屏在数据源异常时显示最近一次成功数据。

依赖关系：

- 依赖 M3。
- 与 M5 强相关，建议在真实接入前完成基础能力。

## 9. M7：一期后段报修接入

目标：

- 在报修系统取数条件具备后接入报修数据。

任务：

- 报修系统适配器。
- 报修统计缓存。
- 今日报修数。
- 处理中报修数。
- 已关闭报修数。
- 平均响应时长。
- TOP 故障设备。
- 最新报修列表。
- 左屏报修占位区替换为真实展示。

验收标准：

- 后台可配置报修系统数据源。
- 可拉取并缓存报修统计。
- 左屏可展示报修数据。
- 报修异常不影响一期前段核心大屏展示。

依赖关系：

- 依赖报修系统接口或数据库条件具备。
- 依赖 M2、M3、M4。
- 优先级低于一期前段核心模块。

## 10. M8：一期后段低优先级 3D 仿真

目标：

- 在报修系统接入之后，再推进 3D 仿真。

任务：

- 3D 模型资源管理。
- 模型展示容器。
- 设备或产线模型展示。
- 展示参数配置。
- 与实时状态联动的预留接口。

验收标准：

- 后台可配置 3D 模型资源。
- 右屏可展示 3D 仿真区域。
- 3D 仿真不影响排产页面性能。
- 未实现实时联动时，可作为静态或半动态展示。

依赖关系：

- 依赖 M4。
- 建议在 M7 后实施。
- 优先级低于报修系统接入。

## 11. M9：二期内部 Web 报表预研

目标：

- 明确权限体系。

- 明确历史数据口径。

验收标准：

## 12. 后台 Schema 维护（M-C）

> 详细字段说明见 **`docs/ADMIN_SCHEMA.md`**。本节定义 **Tier 分级决策** 与 **新增后台页 SOP**，供 M-B antd 重构后的日常扩展使用。

### 12.1 背景与范围

- 19 个侧栏页采用 **schema 驱动 + 共享 CRUD 引擎**（`ResourceCrudPage` / `useResourceCrud`）。
- Schema 源码位于 `frontend/src/admin/schemas/`；构建与 `npm run validate:schemas` 会执行轻量校验（M-C2）。
- 本里程碑 **不重写后端 API**，**不强制 TypeScript**；复杂页允许脱离 schema。

### 12.2 Tier 分级

| Tier | 适用场景 | 实现方式 | 示例 |
|------|----------|----------|------|
| **A** | 标准列表 CRUD：查询、分页、Modal 新建/编辑、批量删除 | 只增 schema + 菜单 + 路由 + 薄 Page | `devices`、`areas` |
| **A+** | Tier A + 引擎已支持的能力（穿梭框、批量工具条、宽 Modal、只读） | 同上，善用 `bulkApplyToolbar`、`screenPageTransfer` 等 | `screenConfigs`、`screenPageBindings` |
| **B** | 列表仍用引擎，但需定制 UI 块（非标准列/表单区） | schema + 引擎扩展点（**M-C4**，未做前尽量避免） | 暂无 |
| **C** | 流程/布局完全特殊，无法描述为字段列表 | 独立 `*Page.jsx`，自行调 API | 未来报表向导、多步配置 |

**升级信号（应离开 Tier A）：**

- 需要在 `ResourceCrudPage` / `useResourceCrud` 中为同一 `resourceKey` 增加 **≥2 处** 硬编码分支；
- 需要向导、分步表单、嵌入图表等非 Modal CRUD 交互；
- schema 字段已无法表达业务，只能靠 JSON 文本「凑」配置。

**默认原则：** 新台账/配置类需求 **先按 Tier A 评估**；不满足再升 Tier B/C，并在 PR 或 `HANDOFF.md` 注明原因。

### 12.3 Tier 决策流程

```text
需要新的后台页？
  │
  ├─ 否 → 只改现有 schema 字段/列/选项（仍 Tier A）
  │
  └─ 是 → 是否为「列表 + 查询 + 分页 + Modal CRUD」？
           │
           ├─ 是 → 现有 field type / cellFormat 是否够用？
           │        ├─ 是 → Tier A（或 A+）
           │        └─ 否 → 新 field 类型？→ 先扩展 schemaRegistry + ResourceField（或等待 M-C4）
           │
           └─ 否 → Tier C（独立 Page）；若仅部分 UI 特殊 → Tier B（M-C4 扩展点）
```

### 12.4 新增 Tier A 页面 SOP

**前置：** 后端 REST 路径与字段命名已确定（响应 camelCase，查询参数 snake_case）。

| 步骤 | 位置 | 动作 |
|------|------|------|
| 1 | 选型 | 确认 Tier A；记录 resourceKey（camelCase，全局唯一） |
| 2 | Schema | 在 `ledger.js` / `screen.js` / `system.js` 或 `dataSources.js` 增加条目；含 `label`、`endpoint`、`columns`、`fields`、`queryFields`；表单末尾 `...RESERVED_FIELDS` |
| 3 | 菜单 | `schemas/menu.js` → `ADMIN_MENU_GROUPS` 对应分组增加 key |
| 4 | 路由键 | `routes/resourcePaths.js` → `ADMIN_RESOURCE_KEYS` 增加同一 key |
| 5 | Lazy 页 | `routes/adminRouteRegistry.js` → `PAGE_LOADERS` 增加 lazy import |
| 6 | Page 文件 | `pages/{Name}Page.jsx` → `export default createResourcePage("resourceKey")` |
| 7 | 校验 | `cd frontend && npm run validate:schemas`（菜单 ↔ 路由 ↔ schema 对齐） |
| 8 | 构建 | `npm run build` |
| 9 | 冒烟 | 登录 → `/admin/{slug}` 深链 → 新建/编辑/删除/查询 |
| 10 | 文档 | 若引入新 field 模式，更新 `docs/ADMIN_SCHEMA.md`；本轮变更写入 `docs/HANDOFF.md` |

**slug 规则：** `resourceKey` 的 camelCase 转 kebab-case（如 `screenConfigs` → `/admin/screen-configs`），由 `resourceKeyToSlug()` 自动生成。

**数据源类页面：** 优先扩展 `dataSources.js` 的 `DATA_SOURCE_TYPE_*`，而非复制整份 schema。

### 12.5 修改现有页 SOP（仍 Tier A）

1. 只改对应 schema 文件中的 `columns` / `fields` / `queryFields` / 选项常量。
2. 若改 `resourceKey` 或菜单结构，同步步骤 3～5。
3. 运行 `validate:schemas` + `build`。
4. 后端字段变更时同步改 API 与 schema 的 `key` 对齐。

### 12.6 验收标准（Tier A 新页）

- `npm run validate:schemas` 无报错。
- `npm run build` 通过。
- 侧栏可进入，URL 可深链刷新，高亮正确。
- CRUD 与列表查询与后端联调通过。
- 未引入引擎内新的 `resourceKey` 硬编码分支（除非已批准升 Tier B/C）。

### 12.7 与 M-C4 / M-C5 的关系

| 里程碑 | 内容 | 不做时的应对 |
|--------|------|--------------|
| M-C4 | 引擎扩展点，注册表驱动 field/column | Tier B 需求暂用 Tier C 独立页，或接受引擎 if 分支 |
| M-C5 | TS、Vitest、三处注册合一 | 靠 `validate:schemas` + 人工同步三处注册 |

**当前状态：** M-C1（拆分）、M-C2（校验）、M-C3（本 SOP）已完成；M-C4/M-C5 按需推进。
